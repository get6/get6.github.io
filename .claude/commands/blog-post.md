# /blog-post - 블로그 글 작성

사용자가 붙여넣은 내용(원문, 메모, 키워드 등)을 토대로 `blog/posts/`에 블로그 게시글을 작성하는 스킬이에요.

## 워크플로우

Obsidian에서 Ctrl+N으로 새 노트를 만들면 Templater가 frontmatter와 Unsplash 이미지를 자동 생성해요.
이 스킬은 **그 이후 단계** — 사용자가 붙여넣은 원문을 블로그 글로 변환하는 역할을 해요.

## 실행 순서

### Step 1: 파일 생성

`blog/posts/` 폴더에 새 마크다운 파일을 생성해요.

**파일명 규칙**: `YYYYMMDDHHmm.md` (현재 시간 기준, zettelkasten ID)

현재 시간은 `date +%Y%m%d%H%M` 명령으로 확인해요.

### Step 2: Frontmatter 작성

```yaml
---
title: <글 제목>
date: YYYY-MM-DD HH:mm
tags:
  - <태그1>
  - <태그2>
---
```

- `title`: 글의 핵심을 담은 간결한 한국어 제목 (15~30자)
- `date`: 파일 생성 시점 (`YYYY-MM-DD HH:mm` 형식)
- `tags`: 글의 주제를 나타내는 소문자 영어 태그 (1~3개)

### Step 3: 대표 이미지

Frontmatter 바로 아래에 Unsplash 랜덤 이미지를 넣어요.

```markdown
![photo by {작가명}({작가 Unsplash URL}) on Unsplash]({이미지 URL})
```

이미지 URL 형식:

```
https://images.unsplash.com/photo-{ID}?crop=entropy&cs=srgb&fm=jpg&ixid={ixid}&ixlib=rb-4.1.0&q=85&w=800&h=460
```

> **참고**: Unsplash API 접근이 안 되면 이미지 부분은 비워두고, Obsidian Templater로 이미지를 추가하도록 안내해요.

### Step 4: 본문 작성

사용자가 붙여넣은 원문을 **독자 관점에서 도움이 되는 블로그 글**로 변환해요.

## 문체 규칙 ("~요" 체)

### 기본 원칙

- **"~요" 체**를 일관되게 사용해요 (존댓말이되 부드럽고 친근하게)
- 독자에게 말을 거는 듯한 톤을 유지해요
- 딱딱한 설명문이 아니라, **친구에게 알려주는 느낌**으로 써요

### 문장 패턴 (실제 블로그에서 추출)

```
좋은 패턴:
- "~할 수 있어요"
- "~하면 돼요"
- "~하는 게 좋아요"
- "~거든요" (이유 설명)
- "~이에요 / ~예요"
- "~해보세요"
- "~해볼게요"
- "~하죠"
- "~잖아요" (공감 유도)

나쁜 패턴 (사용 금지):
- "~합니다 / ~됩니다" (너무 딱딱함)
- "~하시기 바랍니다" (공문서 톤)
- "~하겠습니다" (발표 톤)
- "~인 것이다" (논문 톤)
```

### 도입부 작성법

첫 문단은 `##` 없이 바로 시작해요. 독자의 공감을 끌어내거나 글의 맥락을 간결하게 설명해요.

```markdown
좋은 예:
"요즘 SNS에 짧은 영상 하나 올리는 게 글 하나 쓰는 것보다 반응이 좋을 때가 많죠."
"Java에서 코틀린으로 넘어오는 서비스들이 많죠."
"우리는 종종 '좋은 메모를 많이 남기면 생산성이 높아진다'고 생각해요."

나쁜 예:
"이 글에서는 X에 대해 알아보겠습니다." (너무 형식적)
"X란 무엇인가?" (논문식 도입)
```

## 구조 규칙

### 헤딩 계층

- `##` (h2): 주요 섹션 구분 — 글의 큰 흐름을 나누는 단위
- `###` (h3): 세부 내용 구분 — 섹션 안의 소주제
- `#` (h1): **사용하지 않음** (title이 h1 역할)

### 문단 구성

- 한 문단은 **2~3문장**으로 구성 (4문장 이상이면 분리)
- 문단 사이에 빈 줄 하나로 구분
- 긴 설명보다 **짧은 문장의 연속**이 읽기 편해요

### 강조와 리스트

- **굵은 글씨(`**`)\*\*로 핵심 키워드를 강조
- 나열할 내용은 **번호 목록(1. 2. 3.)** 또는 **불릿 목록(-)** 사용
- 목록 항목에 **굵은 키워드 + 설명** 패턴 사용:

```markdown
1. **훅** (첫 3초) — 시선을 끄는 장면이나 질문으로 시작
2. **하이라이트** — 전달하고 싶은 핵심 내용
3. **CTA** — "구독", "좋아요" 같은 행동 유도
```

### 코드 블록

- 언어 지정 필수 (`kotlin, `typescript 등)
- 코드 앞뒤에 맥락 설명 포함
- 주석은 한국어로

### 표(Table)

- 비교나 정리가 필요할 때만 사용
- 열 수는 3~5개가 적당

### 인용문(Blockquote)

- `>` 로 팁이나 주의사항 강조

## 내용 규칙

### 원문 변환 원칙

1. 붙여넣은 원문의 **핵심 정보를 빠짐없이** 포함
2. 단순 번역/복사가 아니라 **독자가 이해하기 쉽게 재작성**
3. 전문 용어는 **처음 등장할 때 간단히 설명**
4. 사용자가 제공하지 않은 내용을 **임의로 추가하지 않기**

### 원문 유형별 처리

| 원문 유형          | 처리 방법                                  |
| ------------------ | ------------------------------------------ |
| 기술 문서/튜토리얼 | 실용적 예제 코드 포함, 단계별 설명         |
| 에세이/칼럼        | 자연스러운 흐름으로 재구성, 공감 요소 추가 |
| 강의/발표 노트     | 핵심 메시지 추출, 구조화                   |
| 메모/키워드        | 주제를 확장하되 핵심에 집중                |

### 독자 관점 체크리스트

- [ ] 이 글을 처음 보는 사람도 맥락을 이해할 수 있는가?
- [ ] 읽은 후 독자가 **구체적으로 할 수 있는 것**이 있는가?
- [ ] 불필요한 반복 없이 핵심만 전달하는가?
- [ ] 문단이 너무 길어서 읽기 부담스럽지 않은가?

## 마무리 구조

### 본문 마무리

글의 끝은 핵심 요약 또는 다음 행동 제안으로 마무리해요.

```markdown
좋은 예:
"완벽한 영상을 기다리기보다, 지금 가진 영상으로 릴스 하나 만들어보는 게 가장 좋은 시작이에요."
"더 깊이 파고 싶다면 [Kotlin Koans](https://play.kotlinlang.org/koans)로 연습해보세요."
```

### 인용문 (필수)

본문 마지막에 구분선과 영어 명언을 넣어요. 글의 주제와 관련된 영감을 주는 문구를 선택해요.

```markdown
---

> [!quote] {영어 명언}
> — {저자}
```

## 주의사항

- 파일은 반드시 `blog/posts/` 폴더에 생성
- 파일명은 반드시 `YYYYMMDDHHmm.md` 형식
- 이미 같은 파일명이 존재하면 분 단위를 +1 해서 충돌 회피
- 기존 게시글의 톤과 형식을 **일관되게** 유지
- 사용자가 제공하지 않은 내용을 임의로 추가하지 않기
- 이모지 사용하지 않기
